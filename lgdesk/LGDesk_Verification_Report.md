## Accepted Gaps (not fixed this pass)

Documented for the next session ‚Äî each is either blocked on external credentials, a behavioral/product decision, or a broader consistency pass out of scope for a targeted bug-fix round.

- ~~**Meetings Google Calendar/Meet-link integration is a stub returning null**~~ ‚Äî **FIXED this round** (see "Verification Round 2" above). The code path is now wired to reuse `calendar.service.ts`'s authenticated-client pattern and the real `GOOGLE_SERVICE_ACCOUNT_EMAIL`/`GOOGLE_PRIVATE_KEY`/`GOOGLE_CALENDAR_ID` credential vars. It remains **non-functional until real Google credentials are provisioned** (same category as the rest of the Google integration), but is no longer a disconnected stub gated on a phantom env var.
- **Team Tasks / Team Projects data-scope (visibility) model diverges from the "any team member" spec ‚Äî and the two diverge from each other** ‚Äî both `tasks.service.ts` and `projects.service.ts` depart from the spec's flat-Team-match model, but **Projects is measurably more restrictive than Tasks**. Tasks' `visibleToManager` (`tasks.service.ts` ~L331, using `managerScopeIds` ~L322) grants a manager visibility of any task assigned to a subordinate via a subordinate-tree walk (`users.getSubordinateIds`). Projects' `visibleToManager` (`projects.service.ts` ~L192) has **no hierarchical fallback at all** ‚Äî only owner (assigner/assignee) or a direct `assignedTeams` match ‚Äî so a project assigned to a subordinate but not to the manager's own team is invisible to that manager, whereas the equivalent task would be visible. Deferred pending product sign-off since it changes access scope, not a pure bug fix.
- **Missing toast/confirm feedback across approval flows** ‚Äî **FIXED**: the Leave-Approvals, Registrations, and Profile-Requests pages now all have success toasts on Approve/Reject, and all three Approve actions gained a `confirm()` guard (Leave-Approvals and Registrations in Verification Round 2; Profile-Requests in a follow-up parity fix).
